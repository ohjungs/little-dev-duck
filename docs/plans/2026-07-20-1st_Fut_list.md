# LDD 기능 백로그 등록 프롬프트 (Claude Code에 그대로 붙여넣기)

```
CLAUDE.md, DECISIONS.md, Status.md, docs/plans/ 를 먼저 읽고 현재 구조를 파악한다.

[이번 Task: 기능 백로그 등록 — 구현 금지]
아래 기능 목록을 docs/FEATURES.md 로 생성한다. 규칙:

1. 지금은 어떤 기능도 구현하지 않는다. 문서 작업만 한다.
2. FEATURES.md 형식: 카테고리별 표. 열 = 기능명 / 태그(MUST·SHOULD·DEFER·SKIP) / 예상 Phase / 한 줄 사유.
3. 기존 phase 문서와 대조해 이미 확정된 기능은 [기확정]으로 표시하고 중복 등록하지 않는다.
4. MUST와 SHOULD 항목만 기존 14 Phase 로드맵의 어느 Phase에 넣을지 매핑안을 제시한다.
   Phase당 작업량이 50%를 넘지 않게 쪼개고, 필요하면 Phase 추가를 제안한다.
5. DEFER와 SKIP은 백로그에 기록만 하고 로드맵에 넣지 않는다.
6. 매핑안을 보고한 뒤 내 승인을 받기 전까지 phase 문서를 수정하지 않는다.
7. 완료 후 History.md에 이번 Task를 기록한다.

[기능 목록 — 4개 대분류]
FEATURES.md는 아래 대분류 구조로 조직한다:
- 그룹 1 코어 생산성: B(할일), C(습관·집중), D(리츄얼·리포트)
- 그룹 2 워크스페이스·정보: A(워크스페이스 기본기), H(뉴스 브리핑)
- 그룹 3 캐릭터·경험: E(마스코트·게이미피케이션), I(픽셀 오리 오피스)
- 그룹 4 플랫폼 기반: F(개발자 UX), G(제품 완성도), J(보안·프라이버시)
각 카테고리 끝의 "보완 검토" 항목은 리뷰 과정에서 추가된 것으로, 동일하게 표에 포함한다.

- 동적 오피스 레이아웃: 에이전트 수에 따라 책상 슬롯 자동 증설, 임계 인원 초과 시 방 확장·신설(그리드 기반 절차적 배치 규칙, 맵은 정적 파일이 아닌 데이터로 생성) — SHOULD
- 대화 요약에 Gemini 활용(로그를 자연어 브리핑으로): 무료 한도 카운터 공유, 캐시 필수 — DEFER
- 직원 오리에게 지시 내리기(대화창 → Claude Code 세션에 프롬프트 주입, 양방향 제어) — DEFER (읽기 안정화 후)
- 직원 오리 자유 이동(pathfinding), 오리 간 대화 연출, 오피스 가구 커스터마이징, 작업 사운드 — DEFER
- 유사 오픈소스(Pixel Agents류) 사전 조사 후 재활용 검토 — 구현 Phase 첫 Task로 지정
- 미니게임·전투·수집 등 게임 콘텐츠 — SKIP (조작·상호작용은 채택으로 변경됨. 시각화+상호작용까지가 범위)
- [보완 검토] 퇴근 모드: Claude Code 미실행·유휴 시 오리들이 퇴근·수면하는 연출(빈 화면 방지) — SHOULD
- [보완 검토] 이벤트 스로틀링: 초당 이벤트 폭주 시 병합·간추림(렌더링 큐 상한) — SHOULD
- [보완 검토] 픽셀 오피스 리소스 예산: G의 위젯 예산에 포함해 CPU·메모리 상한 측정, 초과 시 프레임레이트 자동 하향 — MUST(설계 규칙)

## J. 보안·프라이버시
- Supabase RLS: 전 테이블 기본 거부(deny-by-default) 정책, 마이그레이션에 정책 포함 — MUST
- 시크릿 관리: service_role 키·API 키 커밋 절대 금지. .env + .gitignore + GitHub secret scanning 활성화 (공개 모노레포 전제라 특히 중요) — MUST
- OAuth 토큰 저장: 위젯은 OS 키체인(Tauri keyring/stronghold), 평문 파일 저장 금지 — MUST
- OAuth scope 최소화: Gmail은 발송(send)과 읽기(read) scope 분리, 각 기능 활성화 시점에 개별 동의 — MUST
- 프롬프트 인젝션 방어: 뉴스 기사·이메일 등 외부 텍스트를 Gemini에 넣을 때 명령으로 해석 금지 원칙, 에이전트의 실행 액션(삭제·발송 등)은 전부 사용자 승인 게이트 통과 — MUST
- 로컬 WebSocket 보안: localhost 바인딩 + 세션 토큰 인증(타 로컬 프로세스 접속 차단) — SHOULD
- Sentry PII 스크러빙: 에러 리포트에 문서 본문·이메일 내용·토큰 미포함 — SHOULD
- 의존성 보안: Dependabot 활성화 + CI에 취약점 검사(pnpm audit) — SHOULD
- 공개 발행 필터: 뉴스레터 웹 발행 페이지에 개인 일정·할일·메모가 섞여 나가지 않게 발행 전 검사 — SHOULD
- 2차 백업(Drive) 시 민감정보 취급 기준 문서화 — DEFER

[무료 운영 원칙 — 모든 설계·구현의 최상위 제약]
1. 유료 결제가 필요한 서비스·API·에셋은 어떤 것도 도입하지 않는다. 유일한 예외는 선택적 도메인 구입(기확정).
2. 확정 스택의 무료 범위 안에서만 설계한다:
   - Vercel Hobby(웹 호스팅·서버리스), Supabase Free(Auth·Postgres·pgvector·Storage·Realtime),
     Gemini Free Tier(요약·임베딩·에이전트), Tauri 2(위젯, 서버 불필요), GitHub(저장소·Actions),
     Sentry Free(에러 리포팅), Kenney CC0(타일셋·사운드), freesound CC0.
3. 각 서비스의 무료 한도 수치는 변동되므로 하드코딩하지 말고, 구현 착수 시 공식 문서에서
   현재 한도를 확인해 docs/CONSTRAINTS_FREE_TIER.md 에 기록·갱신한다. 특히 확인할 것:
   - Supabase Free 프로젝트의 비활성 자동 일시정지 정책 → 방지책(GitHub Actions 주기 ping 등) 설계
   - Gemini Free Tier의 모델별 분당·일일 요청 한도 → 뉴스 요약 "하루 상한"과 큐잉·캐시 설계
   - Vercel Hobby의 cron 개수·주기 제한 → 부족하면 스케줄러는 GitHub Actions schedule로 대체
   - Gmail API 일일 발송 한도(본인 1~2통 발송이므로 여유 예상이나 수치 확인)
4. 비용 게이트: 새 라이브러리·서비스 추가 전 반드시 (a) 무료 여부 (b) 무료 한도 (c) 한도 초과 시
   동작(차단인지 과금인지)을 확인해 보고하고 승인받는다. 과금형 초과는 도입 금지.
5. 무거운 연산(임베딩, 요약)은 캐시 우선: 같은 기사·같은 문서를 두 번 처리하지 않는다.

[구현 상세 설계 지침 — Claude Code 작업 방식]
0. 환경: VS Code + Claude Code 확장. 모든 작업은 모노레포(pnpm + Turborepo) 안에서 수행.
   제안 구조(기존 ARCHITECTURE.md와 충돌 시 기존 문서 우선, 차이점만 보고):
   - apps/web        : Next.js 워크스페이스(BlockNote, 검색, 뉴스 리더)
   - apps/widget     : Tauri 2 위젯(오리 상시 표시, 픽셀 오피스, 알림)
   - packages/core   : 도메인 로직(할일·습관·뉴스 파이프라인), 플랫폼 무관
   - packages/duck   : 오리 상태머신·감정·XP (web과 widget 공용)
   - packages/ui     : 공용 컴포넌트·디자인 토큰(캐릭터 바이블 색상 상수화)
   - supabase/       : 마이그레이션, Edge Functions, RLS 정책
1. 뉴스 파이프라인 구현 순서(각각 독립 Task로 분할):
   (a) RSS 수집기: packages/core에 파서 + 피드 등록 스키마(Supabase) + 중복 판정(URL 정규화·해시)
   (b) 요약기: Gemini 호출 래퍼(재시도·한도 카운터·캐시 테이블), 기사당 3줄 요약+원문 링크만 저장
   (c) 클러스터링: 제목·요약 임베딩(pgvector) 코사인 유사도 기준, 임계값은 설정값으로
   (d) 발송: 스케줄러(GitHub Actions schedule 우선) → 요약 묶음 HTML 메일 → Gmail API 본인 발송
   (e) 위젯 알림: Supabase Realtime 구독 → 오리 신문 연출 → 클릭 시 낭독(웹 TTS는 DEFER)
2. 픽셀 오리 오피스 구현 순서(각각 독립 Task로 분할):
   (a) 사전 조사: Pixel Agents류 유사 오픈소스 존재·라이선스 확인. 재사용 가능하면 포크 검토 후 보고
   (b) 이벤트 소스: Claude Code hooks 설정(PreToolUse/PostToolUse/SubagentStop 등)으로
       {agent_id, role, tool, target_file, status, ts} JSON 이벤트를 로컬 파일 큐에 기록.
       hooks로 부족한 정보는 세션 JSONL 로그 tail 파싱으로 보강 [가정: 커버리지 구현 시 검증]
   (c) 중계: Tauri sidecar(경량 Node 또는 Rust)로 파일 큐 watch → WebSocket으로 위젯에 push
   (d) 렌더링: Canvas 2D부터 시작(PixiJS는 성능 필요 확인 후 도입 — YAGNI), Kenney CC0 오피스 타일맵
   (e) 스프라이트: 직원 오리 시트(대기·타이핑·걷기·대화) + 대장오리 시트(4방향 걷기 사이클 필수,
       구분 요소로 안경 강조 또는 넥타이 등 1개 추가)를 캐릭터 바이블 색상으로 제작.
       제작 경로는 아래 MCP 지침 참조. 32x32 기준, 상태당 2~4프레임
   (f) 상태 매핑 테이블을 packages/duck에 데이터로 정의(코드 분기 아닌 설정): 
       Edit/Write=타이핑, Read/Grep=책장, Bash(test·build)=서버실, 에러=물음표, 유휴=커피
   (g) PDCA 역할별 책상 배치, 클릭=말풍선 요약, 더블클릭=로그 패널
   (h) 플레이어 조작: 그리드 이동(타일 스냅), 충돌 맵, 카메라 팔로우, 키 입력은 위젯 포커스
       상태에서만 활성(백그라운드 시 전역 단축키와 충돌 금지)
   (i) 상호작용 시스템: 인접 타일의 직원 오리 감지 → 상호작용 프롬프트 표시 → 대화창 UI.
       1차 응답은 이벤트 데이터 템플릿(LLM 불필요·무료), "자세히"만 로그 요약(캐시)
   (j) 동적 레이아웃: 레이아웃을 규칙 데이터로 정의(방 크기, 책상 슬롯 좌표 생성 규칙,
       확장 임계값). 에이전트 등록·해제 이벤트에 반응해 맵 재생성, 재생성 중 화면 깜빡임 방지
3. 공통 규칙: 각 Task는 STDD(테스트 먼저), Phase 50% 분할 원칙, 완료 시 History.md 기록.
```
