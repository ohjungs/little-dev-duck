# Phase 15 — 뉴스 브리핑 파이프라인 (수집·요약·발행)

작성일 2026-07-20. Phase 매핑 승인(docs/plans/phase-mapping-proposal-2026-07-20.md, 2026-07-20)에 따른
상세 Task 분해 초안.
목표: RSS·공개 API만으로 뉴스를 수집해 Gemini 3줄 요약으로 가공하고, 중복 기사를 클러스터링한 뒤
데일리 뉴스레터로 본인 Gmail 계정에 자기 발송한다. 발송 결과는 오리 위젯 연출로 알리고, 오리 아침
브리핑(일정+할일+뉴스)과 스크랩→노트 저장으로 워크스페이스와 연결한다. 본문 크롤링·전문 재게시는
저작권 리스크로 설계 전 구간에서 금지한다(articles 테이블은 3줄 요약+원문 링크만 저장, 전문 미저장).
픽셀 오리 오피스(로드맵 Phase 16/17)는 이 Phase의 범위가 아니다.

## 완료 정의 (Definition of Done)

- 등록한 RSS 피드에서 새 기사가 수집되면 url_hash 기준으로 중복 없이 articles에 저장된다
  (전문 미저장, 3줄 요약 + 원문 링크만).
- 같은 기사를 재수집해도 Gemini를 다시 호출하지 않는다(url_hash 캐시 히트). llm_usage 카운터가
  당일 Gemini 호출량을 추적한다.
- 같은 사건을 다루는 기사 2건 이상이 임계값 이상 유사하면 같은 cluster_id로 묶인다.
- 정해진 시각에 GitHub Actions 스케줄러가 그날 수집·요약·클러스터링된 기사를 HTML 뉴스레터로
  조립해 Gmail API로 본인 계정에 1통 발송한다.
- 발송 성공 시 위젯에서 오리가 신문을 들고 알리는 연출이 뜬다. 발송 실패 시 재시도 정책이 동작하고,
  최종 실패는 위젯에 실패로 표시된다.
- 사용자는 온보딩에서 관심 카테고리·구독 키워드를 설정하고, 피드 관리 UI에서 피드를 추가/삭제/
  일시정지/미리보기할 수 있으며, RSS 폴더로 정리하고 키워드로 특정 주제를 음소거할 수 있다.
- 연속 N회 수집에 실패하는 피드는 자동으로 일시정지되고 오리가 알린다.
- 뉴스 리더에서 하이라이트+메모로 스크랩한 기사가 워크스페이스 노트로 저장된다.
- 오리 아침 브리핑에 오늘 일정+할일+뉴스 헤드라인이 함께 전달된다(날씨 낭독을 포함한 전체 "아침
  브리핑"은 D 카테고리 DEFER 항목이라 이번 범위는 일정+할일+뉴스로 한정).
- 전 테이블(feeds/articles/scraps/newsletters/llm_usage 중 사용자 데이터 테이블)에 RLS
  deny-by-default가 적용되고, 마이그레이션마다 down 스크립트가 있다.

## Task 목록 (T1 계약 잠금 -> 이후 파이프라인 순서 의존, T6·T7은 T1 잠금 후 병렬 가능)

### T1. RSS 수집기 + 피드/기사 스키마 (직렬, 계약 잠금)
- [ ] Supabase 마이그레이션(+down): `feeds(id, user_id, url, folder, mute_keywords, status, fail_count)`,
      `articles(id, feed_id, url_hash UQ, title, summary3, cluster_id, published_at)`. 전 테이블
      RLS(user_id = auth.uid()) 포함해 마이그레이션에 함께 커밋
- [ ] packages/core에 RSS 파서 + 피드 등록 인터페이스. 뉴스 소스는 RSS·공개 API만 사용(본문 페이지
      크롤링 금지), 파싱 결과에서 전문 본문 필드는 애초에 들고 오지 않는다
- [ ] 중복 판정: URL 정규화(추적 파라미터 제거 등) 후 SHA-256 해시 -> url_hash UNIQUE 제약으로 DB
      레벨에서도 중복 차단
- [ ] 죽은 피드 감지: 연속 N회(설정값) 수집 실패 시 `status=paused` 자동 전환 + fail_count 리셋 로직
- STDD 명세: 같은 URL을 두 번 수집해도 articles에 1건만 저장(해시 충돌 시 UNIQUE 제약 위반을 애플리케이션이 무시하고 통과). 연속 N회 실패 mock 시 status가 paused로 전환되고, 성공 1회로 fail_count가 리셋되는 단위테스트.
- 계약 잠금 확정(feeds/articles 스키마, RSS 수집 인터페이스). 이후 변경은 병렬 구간 밖에서만.

### T2. 요약기 (Gemini 3줄 요약 + 캐시 + 제목 재작성)
- [ ] `LlmClient` 인터페이스 뒤에 `GeminiClient` 구현(재시도 지수 백오프, `llm_usage(date, kind, count)`
      카운터 테이블 갱신). packages/core는 인터페이스만 의존해 무료 서비스 교체 가능성을 확보(가역성)
- [ ] 기사당 3줄 요약 + 원문 링크만 `articles.summary3`에 저장(전문 재게시 금지). url_hash가 이미
      요약된 기사면 Gemini를 다시 호출하지 않고 캐시 반환
- [ ] 클릭베이트 제목 AI 재작성(사실형 제목으로): 같은 Gemini 호출에 편입해 별도 API 호출을 늘리지
      않는다
- [ ] llm_usage 당일 누적이 설정된 상한에 근접하면 신규 요약을 큐잉하고, 상한 도달 시 원제목 +
      요약 보류 상태로 폴백(무료 한도 초과로 429 받는 상황을 사전 차단)
- STDD 명세: (1) 캐시 히트 시 GeminiClient.summarize가 호출되지 않음을 mock으로 검증. (2) llm_usage가 상한에 도달한 상태를 주입하면 요약 대신 폴백 경로를 타는지 검증. (3) 429 응답 시 지수 백오프 재시도 후 최종 실패를 상위로 전파하는지 검증.

### T3. 중복 기사 클러스터링
- [ ] 제목+요약 텍스트를 Gemini 임베딩(pgvector)으로 변환, 코사인 유사도가 설정값(임계값) 이상인
      기사끼리 같은 `cluster_id`로 묶음. 임계값은 하드코딩하지 않고 설정 데이터로 분리
- [ ] 클러스터 대표 기사 선정 규칙(예: 최초 수집 기사를 대표로) 정의
- STDD 명세: 의도적으로 유사한 요약 2건을 넣으면 같은 cluster_id, 임계값 미만으로 다른 주제 기사를 넣으면 서로 다른 cluster_id가 되는 단위테스트. 임계값을 바꿔도 코드 수정 없이 설정값만으로 재현되는지 검증.

### T4. 발송 (스케줄러 + Gmail 본인 발송 + 실패 재시도)
- [ ] 스케줄러는 Vercel Hobby cron(하루 1회 최소 간격)이 부족하므로 GitHub Actions
      `schedule`(최소 5분 간격, public repo 무료)을 우선 사용해 정해진 시각에 API Route를 호출
- [ ] 그날 수집·요약·클러스터링된 기사를 클러스터 단위로 묶어 HTML 뉴스레터 조립
- [ ] `MailSender` 인터페이스 뒤에 `GmailSender` 구현(재시도 큐 포함). `gmail.send` scope로 본인
      계정에 1통만 발송(대량 발송 금지, SKIP 처리된 항목과 명확히 구분)
- [ ] `newsletters(id, user_id, sent_at, html, status)`에 발송 결과 기록. 발송 실패 시 지수 백오프로
      N회 재시도, 최종 실패 시 `status=failed`로 확정하고 실패 사실을 T5가 위젯에 표시할 수 있게 남김
- STDD 명세: (1) 발송 성공 mock 시 newsletters.status=sent, sent_at 기록 확인. (2) 발송 실패를 연속 주입하면 지수 백오프로 N회 재시도 후 status=failed로 확정되는지 검증. (3) 스케줄러 트리거가 중복 실행돼도 같은 날짜에 뉴스레터가 2통 발송되지 않는 멱등성 검증.

### T5. 위젯 알림 + 오리 아침 브리핑 연계
- [ ] Supabase Realtime으로 `newsletters` INSERT/UPDATE를 구독해 발송 성공 시 오리가 신문을 들고
      알리는 연출을 위젯에 띄움. 클릭 시 낭독(TTS)은 DECISIONS.md 4절대로 차기(Web Speech) 항목이라
      이번 범위에서 제외하고, 클릭 시에는 뉴스레터 본문(요약 목록)을 펼쳐 보여주는 것까지만 구현
- [ ] T4에서 `status=failed`로 확정된 뉴스레터는 위젯에 실패 아이콘/문구로 표시(재시도 진행 중과
      최종 실패를 구분)
- [ ] 오리 아침 브리핑 연계: 정해진 시각에 오늘 일정 + 오늘 할일 + 당일 뉴스레터 헤드라인을 오리가
      말풍선으로 순차 전달. 날씨 낭독 등 전체 "아침 브리핑"(D 카테고리, DEFER)은 포함하지 않는다
- STDD 명세: newsletters INSERT 이벤트를 mock으로 발생시켰을 때 위젯 알림 컴포넌트가 오리 연출을 렌더하는지, status=failed 이벤트에는 실패 UI를 렌더하는지 각각 검증. 아침 브리핑 컴포넌트는 일정 0건/할일 0건/뉴스 0건 조합에서도 에러 없이 빈 상태를 렌더하는지 검증.

### T6. 관심사 온보딩 + 피드 관리 UI (T1 잠금 후 T2~T5와 병렬 가능)
- [ ] 관심사 온보딩: 최초 진입 시 주제 카테고리 선택 + 구독 키워드 등록 플로우
- [ ] 피드 관리 UI: 피드 추가/삭제/일시정지/미리보기(추가 전 최근 기사 몇 건을 미리 보여줌)
- [ ] RSS 폴더 구조(feeds.folder)로 피드를 그룹핑하는 UI, 키워드 음소거(주제 차단) 설정 —
      음소거된 키워드가 제목/요약에 포함된 기사는 뉴스레터·브리핑에서 제외
- [ ] T1의 자동 일시정지(죽은 피드)와 사용자의 수동 일시정지를 같은 status 필드로 통일하되, UI에서
      원인(수동/자동)을 구분해 표시
- STDD 명세: 피드 추가 후 목록에 즉시 반영, 일시정지 토글 시 T1 수집기가 다음 수집 주기에 해당 피드를 건너뛰는 통합테스트. 음소거 키워드를 포함한 기사가 뉴스레터 조립(T4) 대상에서 제외되는지 검증.

### T7. 스크랩 → 워크스페이스 노트 저장 (T1 잠금 후 T2~T5와 병렬 가능)
- [ ] 뉴스 리더에서 기사 본문(요약)에 하이라이트 + 메모를 남기면 `scraps(id, user_id, article_id,
      highlight, memo)`에 저장
- [ ] 스크랩을 워크스페이스 노트(Phase 9에서 확정된 documents/블록 에디터)로 전환하는 액션 제공.
      원문 전체가 아닌 하이라이트+메모+원문 링크만 노트에 옮겨 저작권 문제를 피함
- [ ] 이 Task는 Phase 9(블록 에디터, documents 스키마)가 먼저 서 있어야 실동작한다. 로드맵상
      Phase 15는 Phase 9 이후에 배치돼 있으나(ARCHITECTURE.md 6절), 실제 착수 시점에 Phase 9 완료
      여부를 재확인한다
- STDD 명세: 스크랩 저장 시 scraps에 하이라이트/메모가 정확히 기록되는지, "노트로 저장" 액션 실행 시 documents에 하이라이트+메모+원문 링크만 포함된 새 문서가 생기고 원문 전문은 어디에도 복제되지 않는지 검증.

## 하지 않는 것 (이번 Phase 제외)

- 읽기 큐(나중에 읽기 + 읽음 XP), 키워드 알림(등록 키워드 기사 발생 시 별도 알림), 개인화 추천
  (pgvector 기반), TTS 낭독(오리 목소리), 뉴스레터 전용 수신함(구독 뉴스레터를 앱 안에서 수신),
  키워드 기반 규칙 자동화, 용어 해설/"쉽게 설명" 모드, 같은 사건 매체별 보도 비교, 데일리 퀴즈,
  속보 푸시, 내 큐레이션 매거진 공개 발행(웹 페이지+RSS), 요약 품질 신고 버튼 — 전부 FEATURES.md
  H 카테고리에서 DEFER 처리된 항목이며 이 Phase의 기본 파이프라인이 안정된 뒤 재검토한다.
- 구독자 대상 이메일 대량 발송, 뉴스 사이트 본문 크롤링·전문 아카이빙 — FEATURES.md에서 SKIP
  처리(스팸 정책·저작권 리스크). 어떤 형태로도 이번 Phase에 넣지 않는다.
- 픽셀 오리 오피스(로드맵 Phase 16/17)는 별도 Phase이며 이번 문서에서 다루지 않는다.
- 오리 감정 상태머신·XP 자체의 신규 확장(예: 뉴스 읽음에 대한 XP 지급)은 만들지 않는다. 오리 위젯
  연출은 T5의 "신문 들고 알림"까지만이며, 게임화 로직 변경은 Phase 7의 소관이다.
- 이메일 → 할일 전환, Gmail 어시스턴트 자체(분류/라벨 자동화)는 Phase 10 소관이라 만들지 않는다.
  이번 Phase의 Gmail 연동은 발송(send) 전용이며 읽기(read) scope는 요청하지 않는다.

## 검증 체크리스트 (Phase 종료 시 사용자 실행)

1. RSS 피드 1개 등록 -> 수집 트리거(수동 또는 스케줄) -> articles에 신규 기사 저장, 재수집해도
   중복 없음 확인
2. articles.summary3에 3줄 요약 + 원문 링크만 있고 본문 전문이 어디에도 저장되지 않았는지 확인
3. 같은 기사를 다시 수집·요약 시도 -> Gemini가 재호출되지 않고 캐시 응답, llm_usage 카운트 불변
   확인
4. 유사한 기사 2건 이상 등록 후 클러스터링 실행 -> 같은 cluster_id로 묶이는지 확인
5. 스케줄러(GitHub Actions) 수동 실행 -> 본인 Gmail로 뉴스레터 1통 수신 확인 -> 위젯에서 오리 신문
   연출 확인
6. Gmail 발송을 강제로 실패시켜(예: 인증 만료 재현) 재시도 정책 동작 -> 최종 실패 시 위젯에 실패
   표시 확인
7. 온보딩에서 관심 카테고리·키워드 설정 -> 피드 관리 UI에서 피드 추가/삭제/일시정지/미리보기 동작
   확인
8. 의도적으로 깨진 URL의 피드를 등록해 연속 실패시켜 자동 일시정지 + 오리 알림 확인
9. 키워드 음소거를 설정한 뒤 해당 키워드가 포함된 기사가 뉴스레터·브리핑에서 빠지는지 확인
10. 뉴스 리더에서 기사 하이라이트+메모 스크랩 -> "노트로 저장" -> 워크스페이스 노트에 반영되고
    원문 전문이 복제되지 않았는지 확인
11. 오리 아침 브리핑에 오늘 일정+할일+뉴스 헤드라인이 함께 나오는지 확인
12. 계정 B로 로그인해 계정 A의 feeds/articles/scraps/newsletters가 보이지 않는지 확인(RLS)

## 리스크와 대응

- [추정] Gemini 무료 티어의 실제 RPM/RPD 수치가 CONSTRAINTS_FREE_TIER.md 기준으로 자료 간 편차가
  크다(2.5 Flash RPD 250~1,500 등). llm_usage 카운터는 하한값(예: 250 RPD) 기준으로 보수적으로
  설계하고, 착수 시점에 https://aistudio.google.com/rate-limit 에서 재확인한다.
- [확인됨] Gmail API `messages.send`는 sensitive scope(Restricted 아님)라 CASA 심사는 불필요하지만,
  OAuth Testing 상태에서는 refresh token이 7일 후 자동 만료된다(CONSTRAINTS_FREE_TIER.md). 본인
  1계정 전용 발송이므로 Testing 유지 시 7일마다 재인증이 필요한지, Production 전환(개인 예외
  적용 가능성)으로 회피할지 T4 착수 시 결정한다.
- [확인됨] Vercel Hobby cron은 하루 1회가 최소 간격이라 데일리 뉴스레터 스케줄에는 쓸 수 있어도
  "죽은 피드 감지"처럼 더 잦은 주기의 수집 작업에는 부족할 수 있다. GitHub Actions
  `schedule`(최소 5분 간격, public repo 무료)로 대체하는 것으로 이미 T1/T4에 반영했다.
- [확인됨] Supabase Free는 DB 500MB, Storage 1GB 상한이 있다. articles를 전문 미저장·3줄 요약만
  저장하는 설계로 완화하지만, pgvector 임베딩(T3)이 누적되면 500MB에 먼저 도달할 가능성이 있어
  임베딩 보관 기간(예: N일 이후 정리) 정책을 T3 구현 시 함께 결정해야 한다.
- 클러스터링 임계값(T3)은 초기값이 실측 없이 설정되므로 오탐(서로 다른 사건을 묶음) 또는 미탐
  (같은 사건을 못 묶음) 가능성이 있다. 임계값을 코드가 아닌 설정 데이터로 분리해 배포 없이 조정
  가능하게 했다(T3).
- 죽은 피드 감지(T1)의 "연속 N회 실패" 판정이 일시적 네트워크 장애를 영구 장애로 오판해 정상
  피드를 일시정지시킬 위험이 있다. N값과 재시도 간격을 보수적으로 설정하고 자동 일시정지 사유를
  UI(T6)에 노출해 사용자가 즉시 재개할 수 있게 한다.
- T7(스크랩 → 노트 저장)은 Phase 9의 documents/블록 에디터 스키마에 의존한다. 로드맵상 Phase 15는
  Phase 9 이후에 배치돼 있으나, 실제 착수 시점에 Phase 9가 정말 완료됐는지 재확인이 필요하다
  (아래 착수 조건 참조).
- 저작권 리스크: RSS·공개 API 외 소스(본문 페이지 직접 요청 등)를 어떤 경로로도 추가하지 않도록
  T1의 수집기 인터페이스 자체에서 본문 필드를 다루지 않게 강제했다. 향후 기능 확장 시에도 이
  제약을 재확인해야 한다.
