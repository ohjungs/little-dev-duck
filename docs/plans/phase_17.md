## 완료 정의 (Definition of Done)

- 위젯이 포커스 상태일 때 방향키/WASD로 대장오리가 픽셀 오피스 안을 그리드 단위(타일 스냅)로
  이동하고, 책상·벽 등 충돌 타일은 통과하지 못한다.
- 카메라가 대장오리를 따라가며 오피스 경계 밖을 비추지 않는다.
- 위젯이 포커스를 잃으면(다른 앱으로 전환) 방향키 입력이 오피스 이동으로 소비되지 않고, 동시에
  전역 단축키(Phase 5 Quick Capture 등)는 정상 동작한다.
- 대장오리가 직원 오리 인접 타일에 들어가면 상호작용 프롬프트가 뜨고, 상호작용 키를 누르면
  대화창에 "지금 뭐 하는 중?" 응답이 이벤트 데이터 템플릿(현재 도구·대상 파일·경과 시간)으로
  즉시 표시된다(LLM 호출 없음).
- 에이전트 등록/해제로 임계 인원을 넘거나 밑돌 때 책상 슬롯·방 구성이 규칙 데이터에 따라
  재생성되며, 재생성 중 화면 깜빡임이나 플레이어 위치/카메라 상태 유실이 없다.

## Task 목록 (T1→T2 순서 의존, T3은 Phase 16 이벤트 소스 구독으로 독립적 병렬 가능)

### T1. 플레이어 조작 (그리드 이동, 충돌 맵, 카메라 팔로우)

- [ ] 대장오리 이동 로직을 순수 함수로 분리: `(현재 좌표, 입력 방향, 충돌 맵) → 다음 좌표`
      (타일 스냅, Phase 16 충돌 맵 데이터를 그대로 참조)
- [ ] 카메라 팔로우: 플레이어 타일 좌표를 뷰포트 중심으로 유지, 오피스 경계에서 클램프
- [ ] 키 입력 활성화 조건: 위젯 창이 포커스 상태일 때만 방향키/WASD 리스너를 등록하고, blur
      이벤트 발생 즉시 리스너를 해제한다. 백그라운드 상태에서는 전역 단축키(Phase 5 Quick
      Capture 등)와 절대 충돌하지 않아야 한다 — 이번 Task의 명시적 제약.
- [ ] 4방향 걷기 애니메이션 재생(Phase 16에서 제작된 대장오리 스프라이트 시트 재사용, 신규 에셋
      제작 없음)
- STDD 명세: 이동 순수 함수 유닛 테스트 3케이스(벽 타일=이동 무시, 빈 타일=스냅 이동, 오피스
  경계=카메라 클램프). 포커스 이벤트 테스트: 위젯 blur 시 키 리스너가 해제돼 이동 입력이
  무시되는지, focus 복귀 시 재등록되는지 확인. 전역 단축키 리스너와 방향키 리스너가 동시에
  등록되지 않는지(포커스 상태 배타) 검증하는 케이스 포함.

### T2. 상호작용 시스템 (인접 감지 → 대화창)

- [ ] 인접 타일(상하좌우 1칸) 직원 오리 감지 로직
- [ ] 감지 시 상호작용 프롬프트 UI 노출("상호작용 키를 눌러 대화")
- [ ] 상호작용 키 입력 시 대화창을 열고, Phase 16 상태 매핑 테이블 값(현재 도구·대상 파일·경과
      시간)을 문자열 템플릿에 채워 "지금 뭐 하는 중?" 응답으로 표시. LLM 호출 없이 이벤트
      데이터만 사용하므로 무료이며 지연이 없다.
- STDD 명세: 인접 판정 함수 유닛 테스트(상하좌우 1칸=감지, 대각선/2칸 이상=미감지). 대화창
  응답 템플릿 함수 테스트: 이벤트 데이터(도구/파일/경과시간) 조합별로 문자열이 올바르게
  채워지는지, 이벤트 데이터가 없는 대기 상태 오리는 별도 문구로 표시되는지 확인.

### T3. 동적 레이아웃 (에이전트 수 기반 절차적 배치)

- [ ] 레이아웃 규칙을 설정 데이터로 정의(코드 분기 아님 — ponytail: 설정 데이터 분리): 방 크기
      기준값, 책상 슬롯 좌표 생성 규칙(그리드 간격/오프셋), 확장 임계값(에이전트 수 N명 초과 시
      방 확장·신설)
- [ ] 레이아웃 생성 함수: `(에이전트 목록, 규칙 데이터) → 책상 슬롯 배열` 순수 함수로 구현
- [ ] Phase 16 이벤트 소스의 에이전트 등록/해제 이벤트를 구독해 맵 재생성을 트리거
- [ ] 재생성 중 화면 깜빡임 방지: 기존 렌더를 유지한 채 새 레이아웃을 오프스크린에서 계산 후
      교체하고, 재생성 전후로 플레이어 좌표·카메라 상태를 보존한다.
- STDD 명세: 레이아웃 생성 함수 유닛 테스트 3케이스(임계값 미만/도달/초과)로 슬롯 좌표 중복이
  없는지 확인. 재생성 트리거 통합 테스트: 재생성 전후로 플레이어 좌표·카메라 상태가 동일하게
  유지되는지, 재생성 중 프레임이 끊기지 않고 교체되는지(더블 버퍼링 또는 diff 적용 여부) 확인.
  규칙 데이터를 코드 수정 없이 값만 바꿔 임계값을 조정할 수 있는지 스냅샷 테스트로 확인.

## 하지 않는 것 (이번 Phase 제외)

- 직원 오리에게 지시 내리기(대화창 → Claude Code 세션에 프롬프트 주입, 양방향 제어) — DEFER
  (원문 사유: "읽기 안정화 후"). 이번 Phase는 읽기 전용 상호작용까지만 다룬다.
- 직원 오리 자유 이동(pathfinding), 오리 간 대화 연출, 오피스 가구 커스터마이징, 작업 사운드 —
  DEFER, 부가 연출로 후순위.
- "자세히" 버튼의 로그 요약(Gemini 활용) — DEFER 태그(무료 한도 카운터 공유·캐시 설계가
  선행돼야 함). 이번 Phase에서는 버튼 자체를 포함해 아무것도 만들지 않는다(향후 Phase를 위한
  빈 껍데기 선제 생성 금지 — ponytail). 1차 응답인 이벤트 데이터 템플릿만 T2에서 구현한다.
- 미니게임·전투·수집 등 게임 콘텐츠 — SKIP, 시각화+상호작용까지가 픽셀 오피스의 범위.
- Phase 16 산출물(이벤트 소스, sidecar 중계, 렌더링 엔진, 타일맵·충돌 맵 데이터, 스프라이트
  4종, 상태 매핑 테이블, PDCA 정적 배치, 클릭/더블클릭 상호작용)은 이 Phase의 전제이며 재구현
  하지 않는다.

## 검증 체크리스트 (Phase 종료 시 사용자 실행)

1. 위젯 포커스 상태에서 WASD/방향키로 대장오리가 그리드 단위로 이동하고, 책상·벽 타일은
   통과하지 못하는지 확인
2. 위젯 창 포커스를 다른 앱으로 옮긴 뒤 방향키를 눌러도 대장오리가 움직이지 않고, 동시에
   전역 단축키(Quick Capture 등)가 정상 동작하는지 확인
3. 카메라가 대장오리를 따라가며 오피스 경계 밖을 비추지 않는지 확인
4. 직원 오리 인접 타일에 진입하면 상호작용 프롬프트가 뜨고, 상호작용 키 입력 시 대화창에
   현재 도구·대상 파일·경과 시간이 표시되는지 확인
5. 에이전트 수가 늘어 임계값을 넘을 때 방이 확장/신설되고, 줄어들 때 슬롯이 정리되는 과정에서
   화면 깜빡임 없이 전환되는지 확인
6. 레이아웃 재생성 전후로 대장오리 위치와 카메라 상태가 그대로 유지되는지 확인

## 리스크와 대응

- Phase 16 의존성: 이 Phase는 Phase 16이 만드는 타일맵/충돌 데이터, 이벤트 소스, 상태 매핑
  테이블, 대장오리 스프라이트(4방향 걷기 사이클)가 먼저 확정돼야 시작할 수 있다. [확인됨]
  2026-07-20 기준 phase_16.md 자체가 아직 작성되지 않아, 이 Phase의 실제 착수는 Phase 16의
  계획·구현·검증이 끝난 이후로 순연된다.
- 키 입력 포커스 충돌: 위젯은 항상 최상단(always-on-top, Phase 5)에 떠 있는 상시 실행 앱이라,
  포커스 판정이 부정확하면 사용자가 다른 앱에서 타이핑 중에도 방향키가 오피스 이동으로
  가로채질 위험이 있다. 포커스 상태 판정은 OS/Tauri 창 포커스 이벤트를 단일 신뢰 소스로 삼고,
  포커스 상실 즉시 키 리스너를 해제하는 것으로 완화한다.
- 동적 레이아웃 재생성 중 진행 중이던 대화창·애니메이션 상태가 유실될 가능성 — T3에서 상태
  보존을 STDD 명세에 넣었으나, 실제 프레임 드랍 여부는 구현 후 실측이 필요하다 [가정](+확인
  방법: 재생성을 반복 트리거하며 프레임타임 측정).
- 무료 운영 원칙: 이 Phase 범위는 LLM 호출이 없다(이벤트 데이터 템플릿만 사용). "자세히" 로그
  요약을 DEFER로 미룬 이유이기도 하며, 별도의 무료 한도 리스크는 없다.

## 착수 조건

이 초안은 사용자가 Task 순서/범위를 확인하고 승인해야 착수한다. Phase 매핑 자체는 승인됐으나
(2026-07-20), 이 상세 Task 분해는 별도 승인 없이 구현을 시작하지 않는다. 또한 이 Phase보다
로드맵상 앞선 Phase(3~16)가 아직 시작되지 않았으므로(2026-07-20 기준 Phase 2까지만 진행
중), 실제 착수는 그 이후, 특히 직접 선행 관계인 Phase 16 완료 이후가 된다.
