# Phase 2 — 투두 + 메모 위젯

작성일 2026-07-20, 같은 날 착수·구현 완료(사용자 승인: "착수해").
목표: 위젯 모드의 핵심 두 기능(투두, 메모)을 core 계약(T2에서 잠긴 Todo/Memo)에 맞춰
실제 CRUD로 동작시킨다. 오리 렌더링, 커밋 잔디, 블록 에디터는 이후 Phase다 (로드맵 3/4/9).

## 완료 정의 (Definition of Description)

- 로그인 후 `/`에서 투두 목록을 보고 추가/완료 토글/삭제할 수 있다 (오늘/날짜별 필터 포함).
- 메모를 작성/수정/삭제할 수 있다.
- 빈 상태, 에러 상태, 로딩 상태가 각각 별도 UI로 표시된다.
- 계정 A의 투두/메모는 계정 B에서 보이지 않는다 (Phase 1의 RLS로 이미 보장, 여기선 회귀 확인만).
- 오프라인/네트워크 에러 시 사용자에게 실패가 보이고, 재시도 가능하다.

## Task 목록 (계약 잠금 -> 병렬 2구간)

### T1. packages/api — 투두/메모 데이터 접근 계층 (직렬, 계약 잠금) — 완료
- [x] Supabase 클라이언트는 packages/api로 이전하지 않고, 호출 측이 클라이언트를 주입하는 방식으로
      결정 (packages/api는 `SupabaseClient` 타입만 의존, 인스턴스 생성은 apps/web 소관 유지 —
      Next.js 전용 SSR/브라우저 클라이언트 분기를 packages/api에 끌어들이지 않기 위함. Tauri/RN이
      실제로 붙는 시점에 각자의 클라이언트 인스턴스를 주입하면 되므로 재작업 없음)
- [x] `listTodos`, `createTodo`, `updateTodo`, `deleteTodo` / `listMemos`, `createMemo`, `updateMemo`,
      `deleteMemo` 구현. DB 응답(snake_case)을 core 도메인 타입(camelCase)으로 매핑 후 zod로 검증
- STDD: 목(mock) Supabase 클라이언트로 정상/zod 검증 실패/DB 에러 케이스 13개 테스트, 전부 통과.
- 계약 잠금 확정.

- [x] 빈 상태 / 에러 상태(재시도 버튼) / 로딩 상태

## 검증 체크리스트 (Phase 종료 시 사용자 실행)

1. 투두 추가 -> 새로고침해도 유지 -> 완료 토글 -> 삭제
2. 메모 작성 -> 수정 -> 삭제
3. 네트워크 끊고 액션 시도 -> 에러 상태 노출 확인
4. 계정 B로 로그인해 계정 A의 항목이 안 보이는지 재확인 (RLS 회귀)
