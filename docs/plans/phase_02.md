# Phase 2 — 투두 + 메모 위젯 (초안, 사고 게이트 미통과)

작성일 2026-07-20. Phase 1 종료 시 자동 초안. **아직 사용자 검토/확정 전이므로 착수하지 않는다.**
목표: 위젯 모드의 핵심 두 기능(투두, 메모)을 core 계약(T2에서 잠긴 Todo/Memo)에 맞춰
실제 CRUD로 동작시킨다. 오리 렌더링, 커밋 잔디, 블록 에디터는 이후 Phase다 (로드맵 3/4/9).

## 완료 정의 (Definition of Description)

- 로그인 후 `/`에서 투두 목록을 보고 추가/완료 토글/삭제할 수 있다 (오늘/날짜별 필터 포함).
- 메모를 작성/수정/삭제할 수 있다.
- 빈 상태, 에러 상태, 로딩 상태가 각각 별도 UI로 표시된다.
- 계정 A의 투두/메모는 계정 B에서 보이지 않는다 (Phase 1의 RLS로 이미 보장, 여기선 회귀 확인만).
- 오프라인/네트워크 에러 시 사용자에게 실패가 보이고, 재시도 가능하다.

## Task 목록 (계약 잠금 -> 병렬 2구간)

### T1. packages/api — 투두/메모 데이터 접근 계층 (직렬, 계약 잠금)
- [ ] Supabase 클라이언트 팩토리를 packages/api로 이전 또는 신설 (apps/web뿐 아니라 향후 Tauri/RN도
      packages/api를 통해 데이터에 접근하는 의존 방향 규칙 준수 — [가정] 세부 인터페이스는 착수 시 확정)
- [ ] `listTodos`, `createTodo`, `updateTodo`, `deleteTodo` / `listMemos`, `createMemo`, `updateMemo`,
      `deleteMemo` 함수 시그니처 확정. 반환값은 core의 zod 스키마로 파싱 후 반환 (신뢰 경계에서 검증)
- STDD 명세: Supabase 응답을 목(mock)으로 주입한 단위 테스트 — 정상 CRUD, 잘못된 응답 형태(zod 파싱
  실패) 시 명시적 에러.
- 주의: 이 Task 완료 = API 계약 잠금. 이후 T2/T3는 이 인터페이스에만 의존한다.

### T2. 투두 위젯 UI (T1 이후 병렬 가능)
- [ ] 오늘/날짜별 필터, 완료 토글, 추가/삭제
- [ ] 빈 상태 / 에러 상태 / 로딩 상태 컴포넌트 (packages/ui에 없는 것만 신규, 있으면 재사용)
- STDD 명세: 목록 렌더, 빈 배열 -> 빈 상태 UI, API 에러 -> 에러 상태 UI, 낙관적 업데이트 후 실패 시 롤백.

### T3. 메모 위젯 UI (T1 이후 병렬 가능)
- [ ] 목록/작성/수정/삭제, 빈/에러/로딩 상태
- STDD 명세: T2와 동일 패턴, 메모 본문 10000자 초과 시 프론트에서도 선제 경고(서버는 이미 T2 zod로 방어).

### T4. 위젯 모드 레이아웃 셸 (T2/T3 이후 직렬)
- [ ] `/` 홈을 로그인 확인용 자리표시자에서 투두+메모가 함께 보이는 위젯 대시보드로 교체
- [ ] 오리/커밋 잔디 자리는 빈 슬롯으로만 남기고 구현하지 않음 (선제 껍데기 생성 금지 - ponytail)
- STDD 명세: 반응형 최소 확인(모바일 폭에서 겹침 없음), 로그아웃 버튼 유지.

## 하지 않는 것 (이번 Phase 제외)

오리 렌더링(Phase 3), 커밋 잔디(Phase 4), 뽀모도로/습관/캘린더(Phase 7), 블록 에디터(Phase 9),
알림 4채널, 공개 공유. packages/mascot, ai 디렉터리는 만들지 않는다.

## 검증 체크리스트 (Phase 종료 시 사용자 실행)

1. 투두 추가 -> 새로고침해도 유지 -> 완료 토글 -> 삭제
2. 메모 작성 -> 수정 -> 삭제
3. 네트워크 끊고 액션 시도 -> 에러 상태 노출 확인
4. 계정 B로 로그인해 계정 A의 항목이 안 보이는지 재확인 (RLS 회귀)

## 리스크

- [가정] packages/api의 Supabase 클라이언트 위치(apps/web에서 이전 vs 신규 작성)는 착수 시 실측 후 결정
- 낙관적 업데이트 롤백 로직은 Phase 1에 없던 패턴 — T2/T3 STDD에서 처음 검증
