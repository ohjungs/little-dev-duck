# Phase 16 — 픽셀 오리 오피스 기반 (이벤트·렌더링)

작성일 2026-07-20. Phase 매핑은 docs/plans/phase-mapping-proposal-2026-07-20.md에서 승인됨(2026-07-20).
목표: 플레이어가 직접 조작하지 않아도 완결되는 픽셀 오피스 시각화 인프라를 세운다. Claude Code의
실제 작업(도구 실행)을 이벤트로 뽑아 로컬 WebSocket으로 위젯에 중계하고, Canvas 2D 타일맵 위에서
직원 오리들의 애니메이션 상태로 보여준다. 대장오리(사용자) 플레이어 조작, 직원 오리 상호작용, 동적
레이아웃은 이 Phase에 포함하지 않는다(Phase 17).

이 Phase는 docs/plans/2026-07-20-1st_Fut_list.md "[구현 상세 설계 지침] 2. 픽셀 오리 오피스 구현
순서" (a)~(g) 7단계를 T1~T7 뼈대로 그대로 사용한다. (h)플레이어조작 (i)상호작용시스템 (j)동적레이아웃은
Phase 17로 명시적으로 미룬다.

## 구현 현황 (2026-07-24, `/loop` 인계 세션) — 웹 오피스 렌더링 슬라이스

원래 이 Phase는 apps/desktop(Tauri 위젯 셸) 위에서 동작하도록 설계됐으나, Tauri sidecar WebSocket·
Claude Code hooks·스프라이트 에셋 생성(Adobe/Canva MCP)은 데스크톱/기기/MCP가 필요해 무인 실행에서
검증 불가다. 대신 **웹 앱에서 완결·검증 가능한 렌더링/상호작용/계약 부분**을 구현하고, 실이벤트 소스와
데스크톱 중계는 이월했다:
- **T1 사전조사(완료·결론)**: Canvas 2D + 절차적 스프라이트로 충분히 자체 구현 가능 — 외부 OSS 포크
  불필요(라이선스/아키텍처 결합 리스크 회피). PixiJS 미도입(YAGNI, phase_16 원칙).
- **T2 이벤트 계약(완료)**: core `office-event.ts` — `officeEventSchema`(agentId/role/tool/targetFile/
  status/ts), `parseOfficeEvents`(JSONL, malformed 줄 스킵). 계약 잠금. core 5 tests.
- **T6 상태 매핑(완료)**: core `eventToState`(Edit/Write=typing, Read/Grep=reading, Bash=server,
  error=question, 미지 도구=idle 폴백). 코드 분기 아닌 데이터 테이블. (phase_16은 apps/desktop에 두라
  했으나 웹 구현이라 core에 둠 — 계약이자 테스트 대상.)
- **T4 렌더링(완료)**: web `PixelOffice.tsx` — Canvas 2D 오피스(타일 바닥+책상), 캐릭터 바이블 색상
  (#F6EFDD/#E3D3B9/#A99C65/#352116)으로 오리 절차적 드로잉(외부 스프라이트 에셋 없이), 상태별 아이콘/
  타이핑 bob, 유휴 12초 초과 시 퇴근(offwork) 모드, ~11fps 캡(리소스 예산), prefers-reduced-motion 준수.
- **T7 배치+클릭(완료)**: PDCA 4오리(기획/개발/리뷰/대장) 고정 책상, 오리 클릭 히트테스트→말풍선 활동
  요약(LLM 없이 이벤트 데이터 템플릿). 더블클릭 로그 패널은 이월.
- **데모 구동**: 실 Claude Code 이벤트 대신 시뮬레이터가 OfficeEvent 계약대로 랜덤 이벤트를 만들어
  구동 — 실이벤트는 같은 스키마로 꽂으면 됨. `/office` 라우트 + 네비 추가.
- **이월(데스크톱·기기·MCP)**: T2 실 hooks/JSONL tail 소스, T3 Tauri sidecar WebSocket 중계·토큰
  보안·스로틀링(apps/desktop 필요), T5 스프라이트 시트 에셋(절차적 드로잉으로 대체), T7 더블클릭 로그
  패널, 리소스 예산 자동 프레임레이트 하향의 실측 통합(Phase 5 위젯 예산 확정 후).

## 완료 정의 (Definition of Done)

- Claude Code 세션에서 도구를 실행하면(Edit/Write/Read/Grep/Bash 등) 위젯의 픽셀 오피스 화면에서
  해당 오리의 애니메이션 상태가 실시간으로 바뀐다(타이핑/책장/서버실/물음표/커피 등).
- 오피스 화면에는 기획 오리·개발 오리·리뷰 오리가 고정 책상에 배치돼 있고, 오리를 클릭하면 말풍선
  요약이, 더블클릭하면 로그 패널이 뜬다.
- Claude Code가 종료되거나 일정 시간 유휴 상태면 오리들이 퇴근 모드로 전환된다(빈 화면 없음).
- 세션 토큰 없이 로컬 WebSocket에 접속을 시도하면 거부된다.
- 초당 이벤트가 폭주해도(연속 Edit 등) 화면이 끊기지 않고 병합된 상태로 갱신된다.
- CPU/메모리 사용이 정해진 상한을 넘으면 프레임레이트가 자동으로 낮아진다.
- 직원 오리 4종 스프라이트 시트와 대장오리 4방향 걷기 스프라이트 시트가 캐릭터 바이블 색상
  (#F6EFDD/#E3D3B9/#A99C65/#352116, DECISIONS.md 4절) 대조 검증을 통과해 존재한다. 실제 플레이어
  조작(이 시트를 사용한 이동)은 Phase 17에서 구현한다.

## Task 목록 (T1→T2 직렬로 계약 잠금, 이후 T3/T5/T6은 병렬 가능·상한 2~3, T4/T7은 선행 Task 완료 후)

### T1. 사전 조사 — Pixel Agents류 유사 오픈소스 검토
- [ ] GitHub 코드 검색(`gh search repos`, `gh search code`)으로 "에이전트 활동을 픽셀 오피스/타일맵으로
      시각화"하는 유사 오픈소스 존재 여부 조사
- [ ] 발견 시 라이선스(MIT/Apache 등 재배포·수정 허용 여부) 확인
- [ ] 재사용 가능하면 포크 후보와 통합 방식(이벤트 스키마 어댑팅 범위)을 보고, 라이선스 문제나
      아키텍처 불일치로 재사용 불가면 자체 구현으로 진행 결정을 보고
- [ ] 이 Task는 Phase의 실제 첫 Task로 다른 어떤 Task보다 먼저 실행한다(원문 지시 준수)
- STDD 명세 (조사 Task 예외 — 단위 테스트 대신 산출물 체크리스트로 대체): 이 Task는 코드 산출물이
  없는 조사 Task이므로 단위 테스트 대상이 없다. 대신 검증 가능한
  산출물을 기준으로 삼는다 — 조사 결과가 (a) 후보 저장소명·URL (b) 라이선스명 (c) 재사용 가능/불가
  판단과 근거 (d) 다음 Task(T2 이후)에 대한 영향(포크 시 계획 변경 여부)을 포함한 문서로 남아있는지를
  체크리스트로 확인한다.

### T2. 이벤트 소스 — hooks + JSONL tail 파싱
- [ ] Claude Code hooks(PreToolUse/PostToolUse/SubagentStop 등)를 설정해 도구 실행 전후·서브에이전트
      시작/종료 시점에 `{agent_id, role, tool, target_file, status, ts}` JSON을 로컬 파일 큐(JSONL)에
      append하는 스크립트 작성
- [ ] hooks만으로 커버되지 않는 정보(세션 경계, 장시간 실행 진행 상태 등)는 세션 JSONL 로그 tail
      파싱으로 보강
- [ ] 이벤트 스키마를 zod로 정의하고 문서화 — 이 스키마가 T3 이후 전 Task가 의존하는 계약이므로
      이 Task 완료 시점에 잠근다(이후 변경은 병렬 구간 밖에서만, CLAUDE.md 3-3)
- STDD 명세: hooks payload 픽스처를 입력해 파서가 올바른 이벤트 객체를 큐에 append하는지 단위 테스트.
  필드 누락·malformed JSON 입력 시 크래시 없이 무시하고 로그만 남기는지 테스트.
- 리스크 태그: [가정] hooks가 필요한 필드를 전부 커버하는지는 미확인 — 이 Task 착수 시 실측하고,
  부족하면 JSONL tail 파싱 비중을 늘린다(아래 리스크 섹션 참조).

### T3. 이벤트 중계 — Tauri sidecar 로컬 WebSocket + 보안 + 스로틀링
- [ ] Tauri sidecar(경량 Node 또는 Rust)가 T2의 로컬 파일 큐를 watch해 WebSocket으로 위젯에 push
- [ ] 로컬 WebSocket 보안(J): 127.0.0.1(localhost)에만 바인딩, 연결 시 세션 토큰 검증, 토큰 불일치·
      미제공 연결은 즉시 거부 — 같은 머신의 다른 로컬 프로세스가 임의로 구독하지 못하게 함
- [ ] 이벤트 스로틀링: 초당 이벤트 건수가 임계값을 넘으면 동일 agent_id·tool 이벤트를 병합해 간추린
      뒤 push(렌더 큐 폭주 방지)
- STDD 명세: 토큰 없는/틀린 WebSocket 연결 시도 -> 연결 거부(close) 테스트. 정상 토큰 -> 연결 수립 +
  이벤트 수신 테스트. 초당 N건(임계값 초과) 입력 시 병합 후 실제 push 건수가 상한 이하인지 테스트.

### T4. 렌더링 — Canvas 2D 타일맵 + 퇴근 모드 + 리소스 예산
- [ ] Canvas 2D + Kenney CC0 타일셋으로 오피스 맵 렌더링(PixiJS는 도입하지 않음 — 성능 필요성이
      실측된 뒤에만 도입, YAGNI/ponytail 원칙과 일치)
- [ ] WebSocket으로 수신한 이벤트를 오리 스프라이트 상태 갱신에 반영(상태 매핑 테이블은 T6에서 정의,
      이 Task는 갱신 파이프라인만 구현)
- [ ] 퇴근 모드: Claude Code 미실행 또는 일정 시간 이벤트 없음(유휴) 감지 시 오리들이 퇴근·수면하는
      연출로 전환(빈 화면 방지 안전장치)
- [ ] 픽셀 오피스 리소스 예산: CPU·메모리 사용량 측정, Phase 5에서 확정될 위젯 전체 리소스 예산에
      픽셀 오피스 몫을 통합, 상한 초과 시 프레임레이트를 자동으로 낮춤(크래시·강제 종료 금지)
- STDD 명세: 이벤트 fixture를 넣어 상태 갱신 함수가 올바른 스프라이트 상태로 전이하는지 단위 테스트.
  가짜 타이머로 유휴 임계값 초과 시 퇴근 모드로 전환되는지 테스트. 리소스 측정값이 임계치를 넘으면
  프레임레이트 하향 신호를 내는지 테스트.

### T5. 스프라이트 제작 — 직원 오리 4종 + 대장오리 4방향
- [ ] 직원 오리 시트: 대기·타이핑·걷기·대화 4종, 32x32, 상태당 2~4프레임
- [ ] 대장오리 시트: 4방향(상하좌우) 걷기 사이클, 구분 요소(안경 강조 또는 넥타이 중 1개) 추가,
      32x32, 상태당 2~4프레임 — 실제 조작(Phase 17)보다 에셋 준비가 선행돼야 하므로 이 Phase에서
      함께 제작한다
- [ ] 제작 경로: Adobe/Canva MCP로 초안 생성 → 캐릭터 바이블 색상(#F6EFDD 몸통 / #E3D3B9 음영 /
      #A99C65 부리·발 / #352116 외곽선·안경) 실측값과 픽셀 단위로 대조 검증 → 불일치 시 재생성 또는
      수동 보정 → 최종 채택
- STDD 명세: 완성된 스프라이트 시트에서 사용된 색상 목록을 추출(픽셀 스캔)해 4개 기준 색상 코드와

  수(2~4)를 자동 검증.
- 비용 게이트: Adobe/Canva MCP의 무료 사용 한도는 [가정, 미확인] — 이 Task 착수 시 무료 여부·한도·

## 검증 체크리스트 (Phase 종료 시 사용자 실행)

1. Claude Code에서 파일 편집(Edit/Write) 실행 → 위젯 픽셀 오피스에서 해당 오리가 타이핑 애니메이션
   으로 전환되는지 확인
2. 검색(Grep/Read) 실행 → 책장 상태 전환 확인
3. 빌드/테스트(Bash) 실행 → 서버실 상태 확인, 실패 시 물음표 상태 확인
4. 오리 클릭 → 말풍선 요약 표시 확인, 더블클릭 → 로그 패널 표시 확인
5. Claude Code 세션 종료 후 일정 시간 유휴 → 오리 퇴근·수면 연출 확인
6. 세션 토큰 없이 로컬에서 WebSocket 연결을 시도 → 거부 확인
7. 짧은 시간에 다수 도구 호출을 연속 발생(반복 Edit 등) → 화면 끊김 없이 병합 처리되는지, 리소스
   상한 초과 시 프레임레이트가 자동으로 낮아지는지 확인
8. 직원 오리·대장오리 스프라이트 색상 대조 검증 스크립트 결과가 통과인지 확인
9. PDCA 역할 배치(기획/개발/리뷰 오리)가 고정 위치에 표시되는지 확인

## 리스크와 대응

- [가정] Claude Code hooks(PreToolUse/PostToolUse/SubagentStop)가 이벤트 요구 필드
  (agent_id/role/tool/target_file/status/ts)를 전부 커버하는지 미확인 — T2 착수 시 실측하고, 부족한
  부분은 세션 JSONL tail 파싱으로 보강한다(2026-07-20-1st_Fut_list.md 원문 지침).
- Tauri sidecar의 파일 watch + WebSocket push 지연은 실측 전까지 불명 — Canvas 2D를 우선 채택해
  초기 성능 리스크를 낮춘다.
- **선행 Phase 의존성**: T3(중계)는 Tauri 위젯 셸과 sidecar 인프라(apps/desktop)를 전제로 하는데,
  이는 아직 로드맵상 Phase 5(Tauri 위젯 + Claude Code 수집기)에서 만들어질 예정이며 착수 전이다.
  T4~T7도 위젯 화면 안에서 동작해야 하므로 같은 제약을 받는다. 즉 Phase 16의 실제 구현 착수는 최소
  Phase 5(위젯 셸)가 존재한 뒤에야 가능하다 — Phase 순서상 Phase 3~14가 모두 끝나야 한다는 뜻은
  아니지만, apps/desktop이 없는 상태에서 T3 이후를 시작할 수는 없다.
- 픽셀 오피스 리소스 예산은 G 카테고리의 위젯 전체 리소스 예산(Phase 5에서 정의 예정)과 통합해야
  하는 항목이라, Phase 5의 구체 수치가 확정되기 전에는 픽셀 오피스 몫의 상한값도 잠정치로 둔다.
- 스프라이트 생성 AI 결과물이 캐릭터 바이블 기준색과 크게 벗어나면 재생성 비용이 발생 — 초기
  프롬프트에 정확한 헥스코드를 명시해 리스크를 낮춘다.
- 로컬 WebSocket은 localhost 바인딩만으로는 같은 머신의 다른 로컬 프로세스 접근을 막지 못하므로
  세션 토큰 검증이 필수다 — 토큰 저장·전달 방식은 Phase 5의 OS 키체인(OAuth 토큰 저장) 결정과
  일관되게 설계한다.
- [가정] Adobe/Canva MCP의 무료 사용 한도가 미확인 — T5 착수 시 무료 여부·한도·초과 시 동작을
  확인하고 보고한다. 유료로 확인되면 무료 대안(수동 픽셀아트 제작 등)으로 전환한다.
- 이벤트 스로틀링·리소스 예산 하향 로직이 지나치게 공격적이면 오리 애니메이션이 부자연스럽게
  끊길 수 있음 — 임계값은 잠정치로 시작하고 실사용 검증 후 조정한다.

## 착수 조건

이 초안은 사용자가 Task 순서/범위를 확인하고 승인해야 착수한다. Phase 매핑 자체는 승인됐으나
(2026-07-20), 이 상세 Task 분해는 별도 승인 없이 구현을 시작하지 않는다.

이 Phase보다 로드맵상 앞선 Phase(3~14)가 아직 시작되지 않았다면 실제 착수는 그 이후가 된다. 특히
T3(이벤트 중계) 이후 Task는 Phase 5(Tauri 위젯 + Claude Code 수집기)에서 만들어질 apps/desktop
위젯 셸을 전제로 하므로, 최소 Phase 5가 존재해야 실제 구현이 가능하다(위 리스크 섹션 참조).
